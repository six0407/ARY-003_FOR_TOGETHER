## 2. Git 协作流程

### 2.1 首次拉取

```bash
# 每个成员克隆仓库
git clone <仓库地址> togetherGit
cd togetherGit

# 安装依赖
cd project
npm install
cd ..
```

### 2.2 日常开发流程

```bash
# 1. 开始工作前：拉取最新代码
git pull origin main

# 2. 创建功能分支（按任务编号命名）
git checkout -b dev/你的名字/功能描述
# 示例：git checkout -b dev/张三/DEV-2-public-site

# 3. 在分支上开发、提交
git add .
git commit -m "DEV-2: 完成首页赛事画廊布局"

# 4. 推送到远程
git push origin dev/张三/DEV-2-public-site

# 5. 在 Git 平台上创建 Pull Request，指定 reviewer
```

### 2.3 分支命名规范

| 类型 | 格式 | 示例 |
| --- | --- | --- |
| 功能开发 | `dev/人名/任务编号-简述` | `dev/张三/DEV-2-public-site` |
| Bug 修复 | `fix/人名/简述` | `dev/李四/fix-login-redirect` |
| 文档更新 | `docs/人名/简述` | `dev/王五/docs-permission-update` |

### 2.4 合并规范

- **禁止直接推送到 `main` 分支**
- 每个 PR 至少 1 人 Review 后方可合并
- 合并前确保 `npm test` 通过

## 3. 版本号规则

遵循语义化版本 `主版本.次版本.补丁`：

| 示例 | 含义 |
| --- | --- |
| `v0.1.0` | MVP 初始版本 |
| `v0.2.0` | 新增功能（如完成 DEV-2） |
| `v0.2.1` | Bug 修复或小改动 |

每次发布新版本时：

```bash
# 在 main 分支上打 tag
git tag v0.2.0
git push origin v0.2.0
```

## 4. AgentDocs 使用说明

`AgentDocs/` 目录下的文件是 **AI 上下文指引文档**，供 GitHub Copilot / Claude 等 AI 工具读取项目背景：

| 文件 | AI 读取时机 |
| --- | --- |
| `01-ary-project-analysis.md` | 新成员/新 AI 首次接入时，了解项目全貌 |
| `02-ary-global-design.md` | 涉及架构决策、API 设计时参考 |
| `03-open-questions.md` | 遇到待决策问题时查阅当前讨论议题 |
| `record.md` | 了解历史工作进展和已做决策 |

**注意事项**：
- AgentDocs 是参考文档，**不需要随每次代码提交更新**
- 只有当项目范围、架构或技术选型发生重大变化时才更新
- 日常开发只需关注 `project/` 目录下的代码

## 5. 任务设计文档规范

### 5.1 文档用途

`AgentDocs/devTask/` 下的任务设计文档用于**记录每个开发任务的分析结论和实施方案**，解决"开始写代码前先想清楚"的问题。

### 5.2 何时创建/更新

| 时机 | 操作 |
| --- | --- |
| 开始一个新任务前 | 创建对应的任务设计 `.md`，记录分析结论 |
| 任务范围发生变更 | 更新现有文档的"当前形势"和"任务拆解" |
| 任务完成收口时 | 在文档末尾追加"完成情况"小结 |

### 5.3 文档模板

每个任务设计文档建议包含以下章节：

```markdown
# [任务编号] [任务名]

> 最后更新：YYYY-MM-DD
> 当前阶段：[分析中 / 进行中 / 已完成]
> 前置依赖：[依赖说明]

## 一、当前形势
- 已有资产（已存在哪些代码/接口/设计）
- 当前缺口（还需要做什么）

## 二、任务拆解
列出子任务清单，每个子任务标注：
- 说明
- 涉及的前端页面
- 依赖的后端 API

## 三、设计要点
页面路由、组件划分、数据流等关键设计决策

## 四、验收标准
可量化的完成标准

## 五、完成情况（任务收口时追加）
- [x] 子任务 A 已完成
- [ ] 子任务 B 进行中
```

### 5.4 注意事项

- 任务设计文档是**动态更新的活文档**，不是写死就不动的
- 文档放在 `AgentDocs/devTask/`，与代码在同一仓库，随 PR 一起 Review
- 文档使用中文，保持简洁，不写长篇大论
- 如果有截图/图示能帮助理解，直接放在文档同目录下
